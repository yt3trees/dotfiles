# Fictional Textbook (架空入門書 執筆スキル)

## 背景

架空の入門書をAIに書いてもらい「自分が読む」ことで学ぶ手法。効く理由:

1. **著者の声が立つ** — 本は一人称の視点で貫かれた論旨を要求する。Q&Aでは出ない「主張」「立場」「切り口」が生まれる
2. **構造化が強制される** — 前提→概観→各論→展望の順序が自然に要求される
3. **第N版の効果** — 版数をつけることで改訂を経た「こなれた文体」が要求される
4. **固有名詞の強制** — タイトルやサブタイトルに実在の企業名・プロダクト名を入れることで、抽象論ではなく現実世界の地図になる
5. **比喩で幹を作る** — 一つの強力な比喩が全章を貫くと、読後の印象が段違いに強くなる

ターゲットは **「書店で売っている入門書と同等の密度」**。要約ではなく、実際に読み物として成立する本を作る。

---

## 執筆フロー（必ずこの順で進める）

本を1ターンで書き切ろうとしない。必ず以下の4フェーズを踏む。

### Phase 1: Web検索による情報収集（必須）

以下のいずれかを含む話題なら、書き始める前に `web_search` で最新情報を集める。

- 企業・サービスの名称、価格、プラン、機能、API
- ソフトウェアのバージョン、対応言語、最新のリリース
- 規格・仕様・プロトコルの現時点での状況
- 過去5年以内の出来事
- 人物の肩書き・所属

**検索は浅く済ませない。** 最低でも以下の観点で別々の検索をする:

- 全体像・歴史の把握（例: `GitHub Copilot history Codex`）
- 現在の状態 2025-2026 時点（例: `GitHub Copilot plans pricing 2026`）
- 重要プレイヤー（例: `GitHub Copilot Enterprise features`）
- 技術的な詳細・仕様（例: `Copilot context window FIM`）
- 論点・批判・限界（例: `Copilot security risks studies`）

1トピックにつき5〜10回の検索が目安。`web_fetch` で個別のページを深掘りもする。

**検索しない例外:** 歴史的事実や古典的理論だけで書ける話題（古典力学、江戸時代史、形式言語理論など）のみ。それ以外は必ず検索する。

**推測で具体的な数字を書かない。** 価格・バージョン・統計・日付を推測で書くのは禁止。検索で確認するか、書くこと自体を諦めて抽象的に逃げる（「プランはいくつかの段階に分かれる」など）。

### Phase 2: 章立て設計と合意

検索で得た素材をもとに、章立てを設計する。基本構成:

- **はじめに** — 本を貫く比喩 + 本書の狙い + 扱わない範囲
- **第1章: 前史・前提**（歴史や土台）
- **第2章〜第N章: 各論**（10章前後、各章は 3〜5 個のサブ節を持つ）
- **最終章: 展望**（読者が次に取るべき行動まで踏み込む）
- **おわりに**（短め）

**目次をまずユーザーに提示し、合意を取ってから本文へ進む。** 大きな方向性が違っていたら早めに直すため。ただしユーザーが「任せる」と言えば即進行してよい。

### Phase 3: 本文の反復執筆

**全章を一度に書こうとしない。** Claude の出力には実質的な限界があり、無理に全章を書くと各章が薄くなる。以下の手順で積み上げる:

1. ファイルを作成し、**扉 + 目次 + はじめに + 第1章** まで書く
2. ユーザーに中間成果を見せる（簡潔に）
3. **次のターンで第2章〜第4章** を `str_replace` で追記する
4. **さらに次のターンで第5章〜第7章** を追記する
5. 以下同様に、最終章まで積み上げる

**1ターンで書く分量の目安: 2〜4章（合計 20,000〜30,000字程度）。** それ以上を目指すと品質が落ちる。

ユーザーが「一気に書いて」と言っても、精度のためターンを分ける旨を説明して進める。

### Phase 4: 仕上げ

全章書き終わったら:

- 冒頭の比喩が中盤以降でも参照されているか確認
- ■コラムが十分な数入っているか（10章以上なら最低5本）
- 著者の立場の表明が各章にあるか
- 事実関係に不安のある記述を再度検索で検証

最終ファイルを `/outputs/` に保存、`present_files` で提示。

## 本のフォーマット

### 扉ページ

```
# [トピック]入門

## 【第N版】

[サブタイトル: 固有名詞 3〜5個 + 切り口1フレーズ]

[架空の著者名（絵文字1つ）]
YYYY年MM月  第N版
```

設計のポイント:

- **版数は 2〜5 の範囲で**。第3版あたりが最も落ち着いた雰囲気
- **サブタイトルに必ず固有名詞を入れる**。抽象語だけにしない
- **著者名は造語 + 絵文字1つ**。実在の人名は避ける（例: 「青木テンソル📐」「森ノ下カーネル🧪」）
- **日付は現在の年月**でよい

### 章の内部構造（最重要）

**各章は必ずサブ節に分割する。** これが分量と密度を両立させる鍵。

```
## 第N章: [章タイトル] — [副題]

[章の導入。現実のシーン・問題提起・歴史から入る。1〜2段落]

### N.1 [節タイトル]

[3〜6段落の散文]

### N.2 [節タイトル]

[3〜6段落の散文]

### N.3 [節タイトル]

[3〜6段落の散文]

#### ■コラム [コラム題]

[3〜6段落の脇道深掘り]

### N.4 [節タイトル]

[3〜6段落の散文]

[章末の総括。1〜2段落]
```

**各章の目標字数: 8,000〜12,000字。** これに届かない章は、以下のいずれかで厚みを足す:

- **具体的な事例**を追加（実在の企業・事件・プロダクトの話）
- **歴史エピソード**を追加（いつ誰が何をしたか）
- **コード例や図解**を追加（技術書の場合）
- **業界人の発言・逸話**を追加（検索で見つけたもの）
- **■コラム**を追加

要約的・抽象的な文章で字数を水増しするのは禁止。必ず具体で埋める。

### ■コラムで書きやすいネタ

- 歴史的経緯・命名の裏側
- 業界の逸話・事件
- 規格化の裏話
- 似て非なる技術との比較
- 失敗事例・炎上事件

**本全体で5〜10本のコラムを入れる。** 10章以上の本なら、コラムが1本も無い章があってはならない。

## 分量の目安

| 版 | 総字数 | 章数 | 1章あたり |
|---|---|---|---|
| 簡易版 | 30,000〜50,000字 | 6〜8章 | 4,000〜6,000字 |
| **標準（入門書と同等）** | **100,000〜150,000字** | **12〜15章** | **8,000〜10,000字** |
| 本格版 | 150,000〜250,000字 | 15〜20章 | 10,000〜15,000字 |

**デフォルトは「標準」。** ユーザーが「短めで」「ダイジェストで」と明示した場合のみ簡易版に落とす。

参考: 書店で売っている入門書・新書は 80,000〜200,000字 が一般的。それを下回ると「要約」であって「本」ではない。

## 文体ルール

### やる

- **散文で書く** — 1段落は3〜6文、論理を接続詞でつなぐ
- **著者の立場を表明する** — 「筆者は〜と考えている」「〜と読むのが正確である」のような断定
- **固有名詞を躊躇せず出す** — 実在する企業・製品・規格名・人物名を章の中に埋める
- **比喩を使い回す** — 冒頭で立てた比喩を中盤・終盤でも参照する
- **具体例を豊富に** — 抽象論だけでなく、実際のコード・事件・ケーススタディを入れる
- **漢字を開く** — 「事」→「こと」、「為」→「ため」など自然な日本語に

### やらない

- ❌ 箇条書きの多用（本文は散文で。3項目以上の並列でない限りリストにしない）
- ❌ 「まとめ」「要点」「ポイント」多用
- ❌ 網羅しようとする（入門書は切り口で勝負）
- ❌ AI的な前置き（「以下に説明します」「〜について見ていきましょう」）
- ❌ 絵文字・ビックリマーク（扉の著者名以外）
- ❌ 検索せずに推測で書いた数字・日付・価格
- ❌ 「〜だと言われる」「〜らしい」の多用（出典不明瞭な記述を量産しない）

## 出力形式

1. **デフォルト**: Markdown を `/outputs/[トピック名]入門_第N版.md` に保存
2. 目次を扉の直後に配置
3. 章ごとに区切り線 `---` を入れる
4. 保存後は `present_files` で提示し、短いコメントを添える
5. **PDF化の要求があった場合**: 
   - `python script/convert_to_pdf.py [Markdownのパス]` を実行して PDF を生成する。
   - 生成された PDF も `present_files` で提示する。
   - ※ `markdown2` ライブラリが必要なため、未インストールの場合は指示に従いインストールする。

## 仕上げ前のセルフチェック

出力前に以下を全て確認。足りない項目があれば埋めるまで出力しない。

- [ ] **総字数が目標に達している**（標準なら 100,000字以上）
- [ ] **各章が8,000字以上**
- [ ] **Web検索をして現在の事実関係を確認した**（該当する話題なら）
- [ ] 各章にサブ節 (N.1, N.2, ...) がある
- [ ] ■コラムが本全体で5本以上入っている
- [ ] 扉に「第N版・サブタイトル・固有名詞・著者名・日付」が揃っている
- [ ] はじめにに強い比喩 or 問題提起がある
- [ ] 冒頭の比喩が後半でも参照されている
- [ ] 著者の断定・立場が各所にある
- [ ] AI的な前置き表現が無い
- [ ] 具体的な事例・人物・事件が各章に入っている
- [ ] 検証できない数字や日付を推測で書いていない

## よくある失敗

- **各章が2,000字で終わってしまう** → サブ節に分けていない／具体例が足りない。コラムか事例を足す
- **検索せずに書いた結果、価格や日付が間違っている** → Phase 1 を省略している。必ず検索する
- **一度で全章を書こうとして各章が薄い** → Phase 3 を遵守してターンを分ける
- **要約っぽくなる** → 箇条書きが多いサイン。散文に書き直す
- **比喩が冒頭だけで終わる** → 最終章でも比喩に戻って締めくくる
