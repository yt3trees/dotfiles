# Codex Japanese Style Reference

Codex で `fictional-textbook` を使うときだけ参照する補助メモ。`SKILL.md` 本体の共通ルールを上書きするものではなく、Codex が日本語を不自然にしやすい箇所を補正するための reference。

## 目的

- 比喩が主題を食うのを防ぐ
- 抽象見出しの量産を防ぐ
- 「筆者は」「つまり」「重要なのは」の連打を防ぐ
- 主張先行の硬い評論文ではなく、具体物先行の自然な日本語に寄せる

## 見出し設計

- 各章の設計段階では、素材ノートから「この章で主役にする固有名詞・年号・事件」を先に 3〜5 個抜き出してから見出しを作る
- 章タイトルと節タイトルには、できるだけ 固有名詞 / 年号 / 事件名 / 業界用語 を入れる
- 見出しだけ読んで「誰の、何の、いつの話か」が分かる状態を優先する
- 比喩は副題に置くのは構わないが、主題をぼかしてはいけない
- 「本質」「未来」「変革」「再定義」「入口」「地殻変動」のような抽象語だけで見出しを作らない
- サブタイトルは比喩だけで作らず、必ず具体物を先に置く
- タイトルよりサブタイトルのほうが説明的でよい。雑誌の特集見出しのように「誰が・何を・なぜ」を回収する

悪い例: `OCR市場の本質 — 港は消えない`

良い例: `ABBYYとDX Suite — OCR市場はなぜ2024年以降に再編されたか`

悪い例: `PoCが量産される理由`

良い例: `BCGの74%とPoC止まり — 2024年以降に何が起きたか`

## 段落運び

- まず事実、その次に解釈。主張から入るより、事実から入ったほうが日本語が自然になる
- 1段落目で話題を立て、2段落目で含意を掘り、3段落目で次節へ橋をかける
- 1段落に1つは新情報を置く。同じ主張の言い換えで段落を埋めない
- 抽象語を使うときは、その直前か直後に必ず固有名詞・数字・事件を置く
- 評論文より業界誌特集に寄せる。読者が情景を浮かべられる具体性を優先する

## 文体の抑制

- 比喩は補助線にとどめる。1冊に強い比喩は 1 本まで。なくても成立するなら無理に入れない
- 同じ節で「筆者は」を 2 回以上使うなら、1 回は削るか文型を変える
- 「AはBではなくCだ」のような言い切りは便利だが、連発すると機械臭くなる。1節に1回まで
- 文頭の型を散らす。「筆者は」「つまり」「重要なのは」で段落を始め続けない
- 1文が短すぎると断定が連続して荒く見える。必要なところでは 50〜70字程度まで伸ばしてよい
- 逆に 1文の中に論点を詰め込みすぎない。修飾を重ねるより、文を分けて運ぶ

## 避ける癖

- 比喩を主題より先に立てる見出し
- 固有名詞のない抽象見出しの量産
- 「筆者は」で段落を連打する
- 同じ主張を「つまり」「言い換えれば」「要するに」で3回言い直す
- 一文で気持ちよく締めようとして中身のない格言に逃げる

## NG → 改善例

NG見出し: `生成AIはOCR市場をどう揺らしたか — 港は消えない`

改善: `ChatGPT以後のOCR市場 — ABBYY、DX Suite、Azure Document Intelligenceは何を変えたか`

NG本文: 筆者は、この変化を入口の再定義だと考えている。つまり、OCRはもはや単体の認識技術ではない。重要なのは、業務の入口としての役割である。

改善: 2024年以降、ABBYYは自社を OCR ベンダーより IDP プラットフォームとして前面に出し、AI inside も DX Suite を「AI-OCR」だけではなく後続処理を含むサービスとして売り始めた。ここで起きたのは、文字認識の精度競争が終わったというより、文書入力をどこまで業務フロー全体に接続できるかという競争への移行である。

NG本文: 筆者は、この章でPoCの本質を論じたい。

改善: BCG が 2024 年に示した「生成AIプロジェクトの多くが PoC 後に止まる」という数字は、現場の感覚と大きくずれていない。日本の SIer にとって PoC は失敗ではなく、むしろ受注しやすく責任を限定しやすい契約形態として機能している。
